# Redesign: Answer + Why screens

## The problem in plain words

Today the app gives a great verdict (`GO`) but two things break the experience:

1. **The answer screen doesn't match the question.** The user asked about a 4 PM run, but we show 5 generic hours (2–6 PM) in a small dark strip, then leave the bottom half of the screen blank. We don't know how long they'll be out, so we hedge with a wide window.
2. **The Why screen is text-heavy at the top and empty at the bottom.** Three cards stack near the header, each is mostly prose generated by the AI, the type is small, and the user has to tap each one to discover the real data (charts, stats, timeline).

Underneath both: the three Why cards are written by the AI on each question. They're not "live-bound" to NWS discussions — the AI is *given* AFD + HRRR + radar + SPC as input, then writes its own card titles and descriptions. They refresh every time the user asks a question. That's fine, but the screen doesn't make the data feel *attached* to the answer.

## What we're changing

### 1. Answer screen → "verdict + event card hero"

The full screen becomes about *the user's event*, not a generic 5-hour window.

```text
┌──────────────────────────────────────┐
│ ← BACK              Afton Oaks · 4PM │
│                                      │
│ GO                                   │  ← huge verdict, color-coded
│ Conditions are excellent for         │  ← one calm sentence
│ your 4 PM run.                       │
│                                      │
│ ╔══════════════════════════════════╗ │
│ ║ YOUR RUN WINDOW                  ║ │  ← the hero card
│ ║                                  ║ │
│ ║  3:30 ───── 4:00 ───── 4:30 PM   ║ │  ← only the event hours
│ ║  warm-up    START     finish     ║ │
│ ║                                  ║ │
│ ║  Heat index ▁▂▃▅▅▄ peak 90°F     ║ │  ← sparkline of the metric
│ ║  Rain 2%  ·  Wind 6 mph SSE      ║ │     that actually matters
│ ║                                  ║ │
│ ║  Feels like 89° when you start,  ║ │  ← one line of "what to expect"
│ ║  cools 1° by the time you finish.║ │
│ ╚══════════════════════════════════╝ │
│                                      │
│ One-liner guidance:                  │
│ "Drink before, hydrate at 4:15."     │
│                                      │
│ [ TRACK THIS EVENT ]     [ WHY → ]   │
└──────────────────────────────────────┘
```

Key moves:
- The hourly strip is replaced by a **window built around the event time** (event − 30 min → event + duration + 15 min). For a 1-hour run that's 3 ticks, not 5.
- The metric chosen for the sparkline depends on the verdict driver — heat index here, rain probability for "will it rain", wind for golf, etc. The model already classifies these.
- The verdict, sentence, hero card, and CTA together fill the screen — no dead space.
- Risk meter (HEAT INDEX MANAGEMENT bar) moves *inside* the hero card next to the sparkline; it stops fighting the verdict for attention.

### 2. Event duration: ask once, remember

Right now we don't know how long the user will be out, which is why the timeline shows 5 hours of context. New flow:

- First time a user asks an activity question, we infer a default (running 1h, golf 4h, picnic 2h, wedding 3h) and surface it as an **editable chip** under the question: `Running · 4:00 PM · ~1 hr ✎`.
- Tapping the chip lets them adjust duration. The choice is saved to the user (or device for guests) per activity, so next time "Running" defaults to whatever they picked.
- Duration is what drives the hero card's window, what we send to the model, and what gets stored on a tracked event.

### 3. Why screen → fewer words, more data

Strip the three cards from "prose paragraph + tap to expand" to "headline + visual + one short line." Default to the expanded layout — no tapping required.

```text
┌──────────────────────────────────────┐
│ ← BACK TO ANSWER                     │
│ ATMOSPHERIC CONTEXT · HRRR · 2:24 PM │
│                                      │
│ Your 4 PM run in Afton Oaks is a go. │
│                                      │
│ ── 01 / STABLE PATTERN ──────────────│
│  [☀]  High pressure over Texas       │
│       2%        88°F       51 AQI    │  ← stats grid IS the card
│       rain      high       moderate  │
│                                      │
│ ── 02 / HEAT INDEX ──────────────────│
│  [🌡]  Peaks ~90°F at 5 PM           │
│       ▁▂▃▅▅▄  (HRRR hourly)          │  ← chart IS the card
│       Hydrate before & during.       │
│                                      │
│ ── 03 / RUN WINDOW ──────────────────│
│  [✓]  Locked in: clear, dry, stable  │
│       3:30  4:00  5:00  6:00         │  ← timeline IS the card
│       prep   start  midpt  recover   │
│                                      │
│ Source: NWS HRRR · NWS AFD HOU       │  ← provenance in footer, small
│ Updated 2:24 PM CDT                  │
└──────────────────────────────────────┘
```

Specifics:
- Each card: ~12-word headline + 1 visual (stat row, sparkline, or timeline) + 1 short caption. No paragraph prose.
- Cards are visible by default, not collapsed. The expand-on-tap pattern goes away.
- Source attribution (HRRR, NWS AFD, SPC) and the timestamp move to a single footer line at the bottom — that's what makes the page feel "bound to real data" without each card repeating it.
- Tighter type sizes everywhere; the page should *naturally* reach near the bottom of the viewport instead of hugging the top.

### 4. Make Track the Event meaningful

This becomes the answer to "what do I do with this?" The CTA promises an ongoing weather watch:

- **On tap:** a short confirmation explains what tracking does ("we'll re-check your forecast every hour and ping you only if the verdict changes — GO → CAUTION, rain probability jumps, etc.") so users understand the value instead of guessing.
- **The tracked-event screen** (`/event/$id`) gets a "How the forecast has evolved" timeline showing each snapshot we've taken and what the verdict was, so the user can see the answer get more confident as the event approaches.
- We confirm the existing snapshot cron + push pipeline (`eventSnapshots.functions.ts`, `sendSevereWeatherPush.functions.ts`) is wired to: (a) re-pull the forecast, (b) compare verdict/key-metric to the last snapshot, (c) push only on meaningful change.

## Data sources — answering "where does this come from?"

Documenting it here so the team is on the same page; no code change implied:

| Card / element              | Source                                       | Refresh cadence              |
| --------------------------- | -------------------------------------------- | ---------------------------- |
| Verdict word + sentence     | LLM, given all sources below                 | On each question             |
| Stable pattern / atmosphere | NWS AFD (HOU office in this example)         | ~4×/day, when AFD is updated |
| Heat index / hourly numbers | NWS HRRR via `metDataFetcher`                | Hourly                       |
| Rain probability            | NWS gridpoint forecast                       | Hourly                       |
| Air quality                 | (existing AQI source)                        | Hourly                       |
| Severe / radar trend        | `fetchRadarTrend`, `fetchRotationSignatures` | Every few minutes            |
| SPC outlook                 | `fetchSpcOutlook`                            | Several times/day            |

The three Why cards are **AI-written prose using the above as input**, not direct quotes. Surfacing the source line in the footer makes that relationship visible without re-architecting.

## What lands in this scope

1. New "event card hero" layout on the answer screen, replacing the dark hourly strip.
2. Duration inference + editable chip + storage (`activity → default duration` per user/device).
3. Hero card built around the user's window, with verdict-driven metric (heat index, rain, wind…).
4. Why screen redesigned to "default-expanded, fewer words, more visuals," with a single source/updated footer.
5. Track-the-event explainer + verified push-on-change pipeline, and a forecast-evolution timeline on the tracked-event screen.

## Out of scope (intentionally)

- Changing the underlying weather data sources or fetcher logic.
- Reworking severe-weather / hurricane answer screens (those use their own components — `SevereAnswerScreen`, `HurricaneAnswerScreen`).
- Replacing the AI-generated card prose with rule-based templates. The model writes better adaptive copy than rules would; we're just framing it better.

## Technical notes (for implementation, can skim)

- **Schema:** `WeatherAnswerSchema` already allows signals with `bars`, `stats`, `timeline`, `bar_unit`, `bar_text`. The hero card and the redesigned Why cards reuse these fields; no schema change needed.
- **Prompt:** `systemPrompt.ts` line 140 instructs the model to "always return exactly 3 signal cards." We'll tighten the instructions so card 1 always carries `stats`, card 2 always carries `bars`/sparkline data, card 3 always carries `timeline`. That guarantees the visual variety the new layout depends on.
- **Files touched (likely):** `src/routes/answer.tsx` (hero layout), `src/components/BriefingScreen.tsx` (Why screen restructure), `src/components/SignalCard.tsx` (default-expanded, denser typography), new `src/components/EventWindowCard.tsx` (hero card), `src/lib/systemPrompt.ts` (signal contract), `src/lib/activityDuration.ts` (new — defaults + storage), `src/routes/event.$id.tsx` (forecast-evolution timeline). A small Supabase column may be needed on the user/event for stored duration preferences — flagged for a follow-up migration once we settle the shape.
